# ☁️ Interacting with DVC Remotes

> **Simple Idea:** DVC remotes are like Google Drive for your ML data — a central place where your team stores, shares, and retrieves large data files, while Git only keeps the tiny metadata. `dvc push` uploads, `dvc pull` downloads — just like `git push` and `git pull`.

---

## 🤔 Why Do We Need DVC Remotes?

GitHub (and most Git hosts) have **strict file size limits** — typically 100MB per file. ML datasets can be gigabytes or terabytes. DVC solves this with **remotes** — external storage locations designed for large files.

### Benefits of DVC Remotes

| Benefit | What It Means |
|---|---|
| **Sync large files** | Upload/download datasets and models of any size |
| **Central repository** | Everyone on the team pulls from the same source of truth |
| **Archive versions** | Keep every version of data without filling up local storage |
| **Cloud flexibility** | Works with AWS, GCP, Azure, SSH, and local storage |

---

## 🗄️ Types of DVC Remotes

```
DVC Remotes
│
├── ☁️  Cloud Storage
│   ├── AWS S3

│   └── Azure Blob Storage
│
├── 🖥️  On-Premise
│   ├── SSH server
│   └── HTTP/HTTPS server
│
└── 💻  Local (for testing)
    ├── Local folder on your laptop
    ├── Mounted external drive
    └── Network-Attached Storage (NAS)
```

**Local remotes** are great for:
- Learning and experimenting with DVC
- Testing before setting up cloud storage
- Offline environments

## ⚙️ Setting Up a DVC Remote

All remote settings are stored in `.dvc/config` automatically.

### Add a Remote

```bash
# Add an AWS S3 bucket as a remote named "myAWSremote"
dvc remote add myAWSremote s3://mybucket/data

# Add a Google Cloud Storage bucket
dvc remote add myGCSremote gs://my-gcs-bucket/data

# Add a local folder as a remote (great for testing!)
dvc remote add myLocalRemote /home/user/dvc-storage
```

### Set a Default Remote (with `-d` flag)

```bash
# The -d flag makes this the DEFAULT remote
dvc remote add -d myAWSremote s3://mybucket/data
```

Once a default is set, commands like `dvc push` and `dvc pull` use it automatically — no need to specify it every time.

### Customize Remote Settings

```bash
# Modify settings like region, credentials, etc.
dvc remote modify myAWSremote region us-east-1
dvc remote modify myAWSremote profile my-aws-profile
```

### What Gets Written to `.dvc/config`

```ini
# .dvc/config — auto-generated by DVC
[core]
    remote = myAWSremote         ← this is the default remote

['remote "myAWSremote"']
    url = s3://mybucket/data
    region = us-east-1
```

## 📤 Uploading Data — `dvc push`

After adding files with `dvc add`, upload them to remote storage:

```bash
# Push ALL tracked data to the default remote
dvc push

# Push a specific file only
dvc push data/weather_australia.csv

# Push to a specific (non-default) remote
dvc push -r myGCSremote
```

### What `dvc push` does:
```
Local DVC Cache (.dvc/cache/)
        ↓  dvc push
Remote Storage (S3 / GCS / local folder)
        └── weather_australia.csv  ← uploaded!
```

---

## 📥 Downloading Data — `dvc pull`

When a teammate clones your repo, they get the code and `.dvc` files from Git — but not the actual data. They use `dvc pull` to get it:

```bash
# Pull ALL tracked data from the default remote
dvc pull

# Pull a specific file only
dvc pull data/weather_australia.csv

# Pull from a specific remote
dvc pull -r myAWSremote
```

### What `dvc pull` does:
```
Remote Storage (S3 / GCS / local folder)
        ↓  dvc pull
Local workspace
        └── data/weather_australia.csv  ← downloaded!
```

## 🔄 Git vs DVC Commands — Side by Side

| Action | Git Command | DVC Command |
|---|---|---|
| Upload to remote | `git push` | `dvc push` |
| Download from remote | `git pull` | `dvc pull` |
| Add files to track | `git add` | `dvc add` |
| Set remote location | `git remote add` | `dvc remote add` |
| Fetch without merging | `git fetch` | `dvc fetch` |

## 🔁 Tracking Data Changes — The Full Workflow

When your dataset changes (new rows added, features updated, etc.), follow these 4 steps:

```
Step 1: Re-add the changed file to DVC
        dvc add data/weather_australia.csv
              ↓
        DVC updates the .dvc file with new md5 checksum

Step 2: Stage and commit the updated .dvc file with Git
        git add data/weather_australia.csv.dvc
        git commit -m "Dataset v2.0 - added 2024 records"

Step 3: Push metadata (code + .dvc file) to GitHub
        git push

Step 4: Push the actual new data file to DVC remote
        dvc push
```

### Complete Commands in Order:
```bash
# 1. Re-track the updated data file
dvc add data/weather_australia.csv

# 2. Commit the updated metadata to Git
git add data/weather_australia.csv.dvc
git commit -m "Dataset v2.0 - added 2024 weather records"

# 3. Push code/metadata to GitHub
git push

# 4. Push actual data to DVC remote
dvc push
```

## 👥 Team Collaboration Workflow

Here's how a full team uses DVC remotes day-to-day:

```
Data Scientist A (your machine)          Data Scientist B (their machine)
─────────────────────────────            ──────────────────────────────────
1. dvc add data/weather.csv              1. git clone <repo-url>
2. git commit weather.csv.dvc            2. git pull  ← gets .dvc files
3. git push  → GitHub                    3. dvc pull  ← gets actual data
4. dvc push  → S3 bucket                    └── downloads from S3 bucket

Both are now working with the exact same dataset version! ✅
```

## 💡 Practical ML Example

Suppose your team is working on the weather prediction model:

```bash
# --- Data Scientist A sets everything up ---

git init && dvc init

# Add S3 as default remote
dvc remote add -d weather-data s3://ml-team-bucket/weather

# Track the initial dataset
dvc add data/weather_australia.csv
git add data/weather_australia.csv.dvc .gitignore
git commit -m "Dataset v1.0 - 3 years of weather data"
git push
dvc push    # uploads CSV to S3

# --- 3 months later: dataset is updated ---

# New data added to weather_australia.csv
dvc add data/weather_australia.csv
git add data/weather_australia.csv.dvc
git commit -m "Dataset v2.0 - added 2024 records"
git push
dvc push    # uploads new version to S3

# --- Data Scientist B joins the project ---

git clone https://github.com/team/weather-model
cd weather-model
dvc pull    # downloads the latest dataset from S3 automatically!

# --- Need to reproduce results from 6 months ago? ---

git log --oneline   # find the old commit hash
git checkout abc1234 data/weather_australia.csv.dvc
dvc pull    # downloads the old dataset version from S3!
```

## 🗺️ Summary

| Concept | What It Does | Command |
|---|---|---|
| **DVC Remote** | External storage for large data files | — |
| `dvc remote add` | Register a new remote location | `dvc remote add -d name url` |
| `-d` flag | Set as the default remote | `dvc remote add -d ...` |
| `dvc remote modify` | Customize remote settings | `dvc remote modify name key val` |
| `.dvc/config` | Stores all remote settings | Auto-updated by DVC |
| `dvc push` | Upload data to remote storage | `dvc push` |
| `dvc pull` | Download data from remote storage | `dvc pull` |
| `-r` flag | Target a specific remote | `dvc push -r myRemote` |
| `dvc fetch` | Download to cache (not workspace) | `dvc fetch` |
| **Tracking changes** | Re-add → git commit → git push → dvc push | 4-step process |

---

> ✅ **Key Takeaway:** DVC remotes = cloud storage for your ML data. Set one up with `dvc remote add -d`, upload with `dvc push`, download with `dvc pull`. When data changes, the 4-step update process keeps Git (metadata) and DVC (actual data) in sync. Your whole team can always access the exact same dataset version — no matter where they are.
